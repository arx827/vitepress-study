# 28. 渲染模式 (Rendering modes) 與網站使用體驗核心指標 (Core Web Vitals)
## 前言
  - `Nuxt 3` 作為一個完整的全端框架，兼具開箱即用的特性。

  - 當選擇 `Nuxt 3` 開發網站準備建構正式環境專案時，開發者可以根據使用方式與解決情境，選擇是否啟用伺服器端渲染（`SSR`），或單純產生靜態頁面做 `SSG`。

  - 本篇主要講述 `Nuxt 3` 所提供的渲染模式以及如何進行選擇。

## Nuxt 3 的渲染模式
  - 根據官方文件介紹，`Nuxt 3` 目前支援的渲染模式包含「`僅在客戶端渲染（Client-side Only Rendering）`」與「`通用渲染（Universal Rendering）`」。

  - ### 通用渲染 (Universal Rendering)
    - `Nuxt 3` 預設的配置為：`export default defineNuxtConfig({ ssr: true })`，即預設為 `SSR 通用渲染模式`。

    ```ts
    export default defineNuxtConfig({
      ssr: true
    })
    ```

  - ### 僅在客戶端渲染 (Client-side Only Rendering)
    - 如果想關閉 `SSR`、只在客戶端進行渲染，可將 `nuxt.config.ts` 中的 `ssr` 屬性設為 `false`。

    - 整個專案建構後，運作模式就如同純 Vue 建立出來的 `單頁應用程式（SPA）`，除了首頁的容器頁面，畫面的渲染與響應操作都會在客戶端完成。

## Nuxt 3 的靜態頁面生成 (Static Site Generation, SSG)
  - `Nuxt 3` 在 `v3.0.0-rc.10` 版本實現了「`全靜態生成（Full Static Generation）`」，這才擁有了完全的靜態化，解決了早期版本在静態頁面下仍需發送 API 獲取資料的問題。

  - 透過 `Nuxt` 產生的靜態頁面，會在建構階段將所有 `.vue` 頁面所需的 `JavaScript`、樣式或檔案進行組織與 `預渲染（Pre-rendering）`。

  - 最重要的是，頁面或元件內需要透過 `API` 請求資料的地方，也會一併在建構時取得這些資料，並添加進各自的 `HTML` 頁面與對應的 `Payload` 之中。

## 混合渲染 (Hybrid Rendering)
  - `Nuxt 3` 提供「混合渲染」模式，用以增進各個路由頁面所需的渲染策略。

  - 其概念是允許「`每個路由能有不同的渲染策略`」。例如：文章瀏覽頁面想以靜態頁面（SSG）產生，而管理後台只需要客戶端渲染（CSR）。開發者只需透過指定路由及設定屬性，就能在建構時自動完成自訂的策略並生成頁面。

  - `Nuxt` 官方預期在 `nuxt.config` 提供 `routeRules` 來設定不同頁面的渲染策略，例如：
    ```ts
    export default defineNuxtConfig({
      routeRules: {
        '/': { prerender: true }, // 每一次建構時，都重新預渲染頁面
      }
    })
    ```

## 網站使用體驗核心指標 (Core Web Vitals)
  - ### Largest Contentful Paint (LCP) 最大內容繪製
    主要用以衡量網頁的「`載入速度`」。當進入網站時，網頁會開始載入文字、圖片等並繪製畫面元素。當可視區域不再有新的、更大的元素載入後，LCP 時間才截止。若使用者等待大圖的時間過長，容易導致體驗不佳。建議控制在開始載入後的 `2.5 秒內`。
  - ### First Input Delay (FID) 首次輸入延遲
    主要用以衡量使用者能在網站上進行操作的時間，也稱為「`網站可進行互動的時間`」。當使用者點擊連結、按鈕而驅動了 JavaScript 操作事件，直到瀏覽器做出回應並能實際處理事件所經過的時間即為 FID。為了良好體驗，建議控制在 `100 毫秒以內`。
  - ### Cumulative Layout Shift (CLS) 累計布局排版偏移
    衡量網頁布局排版的「`視覺穩定性`」。若使用者在網頁上點擊物件時，因為某些原因跳出新元素導致物件位置被擠壓、發生位移，會造成使用者誤觸非預期位置。此數值建議應控制在位移比例 `低於 0.1 秒內`。

### Nuxt 3 渲染模式與網站使用體驗核心指標
  - 在核心指標中，`LCP（最大內容繪製）` 與 `FID（首次輸入延遲）` 這兩項指標，與 `Nuxt 3` 渲染模式的選擇最為相關。
  - 選擇 `SSR` 不光只是為了解決 `SEO` 的問題，還需要考量使用者的體驗。
    - 例如：選擇「通用渲染」來製作 SSR + SPA，雖然解決了 SEO 且讓網頁第一次就能完整顯示出來，但在 Vue 的 JavaScript 載入之前，網站其實是不具有互動性的，必須直至 `Hydration（客戶端活化/注水）` 步驟完成，使用者才能操作網頁上的元件。
  - 這種選擇會直接引發 `首次內容繪製（FCP）` 與 `可互動時間（TTI）`的平衡問題，而 `LCP` 與 `FID` 也通常發生在這之間。
  - 因此，`SSR` + `SPA` 並非萬用解，開發者仍需因應各種情境搭配使用 `SSG`、`ISR` 等不同的渲染方式。`Nuxt 3` 幫忙應付了多數可能遇到的使用情境與痛點，開發者可依據需求靈活調整各頁面或整站的渲染方式。

### 小結
  - 本篇總結了 `Nuxt 3` 支援的各種渲染模式（`CSR`、`Universal、Hybrid`）及其特點。
  - 開發者在佈署正式專案時，不應盲目選擇單一模式，而應結合網站使用體驗核心指標（Core Web Vitals）的考量，針對不同類型的路由（如前台文章與後台管理）配置最合適的渲染策略，以兼顧 `SEO`、`載入速度` 與 `互動流暢度`。
