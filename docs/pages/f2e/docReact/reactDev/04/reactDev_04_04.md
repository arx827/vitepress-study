# 調適和故障排除
  本指南可幫助你在使用 React 編譯器（`React Compiler`）時識別和修復問題。學習如何調試編譯問題並解決常見錯誤。

  :::info 你將學到
  - 編譯器錯誤與運行時問題的區別
  - 導致編譯失敗的常見模式
  - 分步調試工作流程
  :::

## 了解編譯器行為
  React 編譯器旨在處理遵循 [React 規則](https://zh-hans.react.dev/reference/rules) 的代碼。當遇到可能違反這些規則的代碼時，它會安全地跳過優化，而不是冒險改變應用程式的行為。

  - ### 編譯器錯誤與運行時問題
    - #### 編譯錯誤
      發生在建構時，並會阻止你的代碼編譯。這類錯誤很少見，因為編譯器的設計理念是跳過有問題的代碼，而不是直接失敗。

    - #### 運行時問題
      發生在已編譯的代碼行為與預期不符時。如果你遇到了與 `React` 編譯器相關的問題，通常是運行時問題。這往往是因為你的代碼以編譯器無法檢測到的方式輕微違反了 React 的規則，導致編譯器錯誤地編譯了一個本應跳過的組件。

    - #### 調試核心思維
      在調試運行時問題時，應集中精力在受影響的組件中查找 `ESLint` 規則未能檢測到的 `React` 規則違規情況。

## 常見的破壞性模式
  `React Compiler` 可能導致你的應用出錯的一個主要方式是：如果你的代碼是基於「`正確性`」而非單純的「`效能優化`」來編寫手動記憶化（`memoization`）。這意味著你的應用依賴於某些特定值的記憶化才能正常運作。

  由於編譯器的記憶化方式可能與你手動實現的方式不同，這可能導致一些意外行為（例如副作用重複觸發、無限迴圈或更新遺失）。常見場景包括：
  - `依賴引用相等性的副作用`：當副作用依賴於對象或陣列在多次渲染中保持相同的引用。
  - `需要穩定引用的依賴陣列`：當不穩定的依賴導致副作用觸發過於頻繁或產生無限迴圈。
  - `基於引用判斷的條件邏輯`：當代碼使用引用相等性檢查來進行快取或優化。

## 調試工作流程
  - ### 遇到編譯器建構錯誤（Build Errors）
    如果你遇到一個意外中斷建構的編譯器錯誤，這很可能是編譯器中的一個 `bug`。請將以下信息報告到 [facebook/react](https://github.com/facebook/react/issues) 倉庫：
    - 錯誤信息
    - 導致錯誤的源代碼
    - 你使用的 React 和編譯器版本

  - ### 遇到運行時問題（Runtime Issues）
    關於運行時行為問題，請依序執行以下分步工作流程：

  - ### 1. 臨時禁用編譯
    使用 `"use no memo"` 指令來確定問題是否與編譯器相關：

    ```js
    function ProblematicComponent() {
      "use no memo"; // 跳過該組件的編譯優化
      // ... 其餘的組件代碼
    }
    ```

    - `測試判斷 A`：如果加上指令後問題消失，則很可能與違反 `React 規則` 有關。
    - `測試判斷 B`：你還可以嘗試從問題組件中移除所有手動的 `memoization`（`useMemo`、`useCallback` 和 `memo`），以驗證在沒有任何快取的狀況下應用程式是否能正常工作。如果移除後問題仍然存在，則說明該組件存在必須修復的 React 規則違反情況。

- ### 2. 逐步修復問題
  - 識別根本原因（通常是上述為了「正確性」而寫的記憶化問題）。
  - 每次修復後進行測試。
  - 修復完成並確認正常後，移除 `"use no memo"` 指令。
  - 在 `React DevTools` 中驗證組件是否重新顯示 `✨` 徽章。

## 報告編譯器錯誤
  如果你認為發現了一個編譯器本身的 `bug`，請依序確認並提交：
  - `確認不是違反 React 規則的問題` - 先使用 ESLint 進行完整檢查。
  - `創建最小的複現代碼` - 在一個乾淨的小示例中隔離該問題。
  - `在不使用編譯器的情況下測試` - 確認該問題僅在開啟編譯器時才會出現。
  - `提交一個 GitHub Issue`，並包含：
    - React 和編譯器的具體版本。
    - 最小複現代碼（Minimal Reproducible Example）。
    - 預期行為與實際行為的差異對比。
    - 任何相關的錯誤日誌或信息。
